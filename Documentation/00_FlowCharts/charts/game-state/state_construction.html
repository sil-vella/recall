State construction

How match state is constructed and updated: backend stores root + game_state per room. Updates are validated at the root level only (StateQueueValidator): each key must be in the schema, values checked by type and allowed/range; no deep validation of nested game_state or games. Then merge and broadcast; frontend receives game_state_updated, adds or updates the game in the games map and syncs widget slices.

Initial match state is created at the “Build / update game_state map” step in _handleStartMatch (game_event_coordinator.dart).

Full example of state from root (single roomId) after _handleStartMatch. Comments indicate method/file that initially creates each key (backend: dart_bkend_base_01).

Root: game_id, game_state from store; roomDifficulty from dutch_game_main._onRoomCreated; turn_events from game_registry; isClearAndCollect at root from message_handler (random join). Tournament: is_tournament and tournament_data (id, name, format, leaderboard, matches_left) from create_room/join_room event data, set on game_state when building match for tournament flow.
